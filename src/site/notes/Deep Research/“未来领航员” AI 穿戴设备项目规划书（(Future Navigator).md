# “未来领航员” AI 穿戴设备项目规划书 

## 1. Why（为什么做：多维目标与核心痛点）

### 1.1 教育痛点（根本价值：回归教育本质）

- **突破“注意力黑洞”：** 在场景化教学（如户外探索、厨房烘焙）中，一位老师通常需要面对 5-10 个孩子。老师的精力上限决定了 80% 以上的孩子间互动、发音细节和行为闪光点会被遗漏。设备将充当“永不疲倦的观察者”，捕获所有流失的教学数据。
    
- **破解“反馈滞后”：** 传统的学习反馈依赖老师的模糊印象，往往只有“表现不错”等笼统评价。本项目通过 AI 能够给出精准到每个音节的发音建议和词汇使用分析，实现真正的“千人千面”个性化教学。
    
- **强化“安全边界”：** 场景化教育往往伴随复杂环境。通过视觉 AI 识别河边、马路、冲突等风险，能弥补人力看管的死角，降低机构运行风险。
    
- **证据化教学效果（可视化）：** 解决家长对“玩中学”是否有用的疑虑，将“玩”的过程转化为具有科技感、数据驱动的成长报告。
    

### 1.2 技术探索目标（创新驱动）

- **验证 Gemini 3 多模态能力：** 探索 Gemini 在处理“超长上下文音频+视频轨迹”时的深度理解能力，特别是在嘈杂教学环境下的语音提取与多语义理解。
    
- **边缘与云端协同架构：** 验证“穿戴端轻量化采集 + 云端大模型深度分析”的技术链路在教育垂直领域的延迟表现与能效比。
    

### 1.3 竞赛目标（短期里程碑）

- **Google 黑客松冲奖：** 充分利用 Gemini 3 的原生多模态特性，打造一个具备硬件闭环、解决真实社会痛点（儿童教育与安全）的高完成度作品。
    
- **技术社区影响力：** 展示 Gemini 在非通用领域（场景化外语习得）的深度应用潜力。
    

### 1.4 商业潜力（长期愿景）

- **教培技术壁垒：** 作为“未来领航员”项目的核心技术护城河，从“卖课程”转向“卖数据服务+智能硬件”。
    
- **订阅制 SaaS 模型：** 探索“设备租售 + 个人成长数据分析月费”的商业模式。
    

## 2. What（项目是什么：产品定义与核心功能）

### 2.1 产品形态定义

- **设备名称：** Navigator-Link (领航员之环)
    
- **物理形态：** 初步设想为 **“多功能智能胸卡”**（挂绳式）或 **“头戴式轻量麦克风”**。
    
    - _选择理由：_ 孩子在活动中手部需进行操作（如烘焙），胸卡形态能提供第一视角或近场视角，且麦克风距离声源较近，干扰小。
        

### 2.2 核心功能清单

#### A. 多模态数据采集（感知层）

- **高清拾音系统：** 采用阵列麦克风，配合基础降噪，确保在嘈杂的工作坊环境中清晰捕捉孩子对话。
    
- **环境视觉传感器：** 广角镜头记录活动关键帧，用于分析孩子与环境的互动（如是否在操作烤箱、是否与同伴对视）。
    
- **IMU 与定位：** 传感器辅助判断运动状态，GPS/蓝牙信标用于户外围栏预警。
    

#### B. 云端 AI 分析引擎（核心层 - 基于 Gemini）

- **语音解析（Speech Analysis）：**
    
    - **发音纠偏：** 自动识别 Phoneme 级别的发音错误，生成本节课“待强化单词表”。
        
    - **表达深度：** 分析句子复杂度（MLU）、词汇丰富度，判断孩子是否在“舒适区”之外尝试新表达。
        
- **行为与社交洞察（Behavioral Insights）：**
    
    - 识别社交角色（如附件 OKR 中提到的：协作角色倾向、领导力体现）。
        
    - 情绪分析：捕捉孩子在遇到困难（如面粉撒了）时的情绪波动及应对方式。
        
- **安全行为审计：** 实时检测危险动作或异常环境距离。
    

#### C. 交付成果（输出层）

- **教师端 App：** 课后即时收到全班发音共性问题汇总，课堂实时接收危险告警。
    
- **家长端“AI 动态报告”：** 包含孩子课堂精彩瞬间（视频/音频剪辑）、多维度雷达图（语言、协作、逻辑、勇气）、AI 建议的课后互动练习。
    
- **阶段性汇总报告：** 利用 Gemini 超长上下文，将 10 节课的数据进行纵向对比，生成季度成长白皮书。
    

### 2.3 MVP（最小可行性产品）范围

- 硬件：基于树莓派/ESP32 的原型设备，实现稳定音视频推流/暂存。
    
- 软件：跑通“采集 -> Gemini 分析 -> 结构化数据存入 Firestore -> Web/H5 报告生成”的全链路。

## 3. Who（为谁而做：用户与干系人）

### 3.1 核心用户

- **孩子（体验核心）：** 3-12 岁儿童。设备对他们而言应当是“隐形”的，通过无感采集让他们摆脱“为了录音而说话”的压力，在自然状态下获得安全保障和能力评估。
    
- **老师/Facilitator（价值核心）：** 课程的导演者。设备不仅是记录工具，更是他们的“第二双眼”和“超级助教”，将他们从繁重的笔记记录中解放出来，专注于引导互动。
    

### 3.2 关键客户

- **家长（付费核心）：** 对教育效果有显性化需求。通过极具科技感的 AI 报告（可视化数据、高光时刻片段），直观看到孩子的点滴成长，缓解教育焦虑。
    
- **教育机构（采购方）：** 如“未来领航员”工作坊。通过配备该系统，提升课程的科技含金量，形成差异化竞争优势，同时大幅降低安全责任风险。
    

### 3.3 干系人

- **项目执行方（开发者）：** 全栈统筹硬件、云端逻辑与 Gemini 模型调优，确保链路通畅。
    
- **竞赛评委（评审方）：** 关注 Gemini 3 的技术应用深度（如多模态融合、长上下文处理）、产品完整度以及对解决社会实际问题（教育不平等、儿童安全）的贡献。
    

## 4. When（时间节奏：开发与竞赛里程碑）

### 4.1 总体节奏

- **总时长：** 约 7-8 周（紧密绑定 Google 黑客松赛程）。
    

### 4.2 关键里程碑

- **M1: 概念设计与 AI 管道验证（Week 1-2）**
    
    - 确定 Gemini API（1.5 Pro/Flash）接入方案，跑通音视频文件的离线上传与分析。
        
    - 编写初步 Prompt，测试其对“烘焙”、“超市”等特定场景对话的识别准确度。
        
- **M2: 硬件原型与系统集成（Week 3-5）**
    
    - **硬件端：** 使用 Raspberry Pi 或 ESP32-CAM 搭建原型，实现音视频本地暂存与自动上传云端。
        
    - **后端：** 基于 GCP 构建数据存储与分析流。
        
    - **教师/家长端：** 开发简单的 H5 页面用于展示报告和实时警报。
        
- **M3: 场景内测与算法优化（Week 6）**
    
    - 在模拟工作坊（如家庭烘焙场景）中实测设备。
        
    - 根据实测噪音和光线情况，优化拾音算法和 Gemini 的 Prompt 策略（解决多孩混响问题）。
        
- **M4: 包装提交与答辩准备（Week 7-8）**
    
    - 制作 3 分钟演示视频（包括穿戴展示、AI 分析过程图、最终生成的成长报告）。
        
    - 整理代码库、技术文档，准备黑客松最终答辩。

## 5. Where（实施环境：技术栈与应用场景）

### 5.1 技术实施环境 (Technical Stack)

本项目将构建一个“端-云-端”的闭环架构，充分发挥 Google 生态的集成优势：

- **核心大模型 (LLM/LMM)：**
    
    - **Google Gemini 1.5 Pro/Flash：** 核心引擎。利用其**原生多模态**能力处理视频帧与音频流，利用其**百万级长上下文**对比孩子在不同课时的表现，生成阶段性报告。
        
    - **实验阶段：** 辅助使用通义千问等国内模型进行基础文本 Prompt 的快速调优。
        
- **硬件原型 (Edge Device)：**
    
    - **开发板：** 树莓派 (Raspberry Pi 4B/5) 或 ESP32-S3（视体积需求而定）。
        
    - **传感器：** 高灵敏度麦克风阵列（如 ReSpeaker）、广角摄像头模块、锂电池供电管理。
        
- **云端服务 (Google Cloud Platform)：**
    
    - **数据存储：** Cloud Storage 用于存储音视频源文件。
        
    - **数据库：** **Firestore** 用于存储结构化的分析结果、学生画像及报警记录（遵循项目要求的存储路径）。
        
    - **计算：** Cloud Functions 触发器，实现“上传即分析”。
        
- **交互终端：**
    
    - **前端框架：** React 或 HTML5 响应式 Web 应用（便于在各类移动端查看报告）。
        

### 5.2 产品应用场景 (Application Scenarios)

- **“未来领航员”工作坊：** 
    - **室内烘焙 (Baking)：**  监测工具使用安全，记录“搅拌”、“加入”等动作关联的英语表达。
    
    - **社区超市 (Grocery)：** 户外弱网络环境下的离线数据暂存，课后上传分析采购时的对话流利度。
        
    - **公园探索 (Park)：** 重点开启“地理围栏”与危险环境预警。
        

## 6. How（如何实现：行动路径与资源）

### 6.1 具体行动步骤 (Action Steps)

#### 第一阶段：AI 核心管道构建（最优先级）

1. **数据模拟：** 抛开硬件，先用手机在工作坊现场录制 15 分钟的音视频，作为测试集。
    
2. **Prompt 工程：** 针对 Gemini 编写多模态 System Instruction。
    
    - _任务 A：_ 提取音频中的发音错误并与标准音对比。
        
    - _任务 B：_ 识别视频中的危险动作（如手靠近火源）。
        
3. **管道打通：** 实现 `模拟数据 -> Gemini API -> 结构化 JSON 报告` 的全流程。
    

#### 第二阶段：硬件原型开发

1. **采集方案：** 编写 Python 脚本实现音视频的分片录制与本地加密。
    
2. **同步机制：** 实现设备连接 Wi-Fi 后，自动将缓存数据同步至 Cloud Storage。
    
3. **佩戴设计：** 3D 打印一个简易的胸卡外壳，解决摄像头俯仰角问题。
    

#### 第三阶段：系统全链路集成

1. **后端开发：** 部署 Cloud Functions，当 Storage 有新文件时，自动调用 Gemini 进行处理并存入 Firestore。
    
2. **UI 开发：** 制作家长/老师端 Web 后台，实现数据的可视化展示（雷达图、关键词云）。
    

#### 第四阶段：竞赛准备与冲刺

1. **实测迭代：** 在一次真实的“烘焙工作坊”中全员佩戴，收集真实噪音背景下的数据，调优 Prompt。
    
2. **作品包装：** 录制演示视频，突出 Gemini 如何在复杂的协作场景中精准识别每个孩子的贡献。
    

### 6.2 所需资源与依赖 (Resources & Dependencies)

- **人力资源：** 需要具备全栈开发能力（Python/Node.js）、基础硬件焊接/调试能力、以及深度的 Prompt Tuning 经验。
    
- **物力资源：** 硬件套件（约 $100-$200 预算）。
    
    - GCP 账号及 Gemini API 调用额度。
        
- **关键依赖：** 
    - **API 稳定性：** 确保在黑客松期间 Gemini 接口的响应速度满足生成实时预警的需求。
    
    - **数据质量：** 嘈杂环境下的降噪效果是 AI 分析准确率的决定性因素。
        

## 7. 结语：项目的社会价值与愿景

本项目不仅是一个技术竞赛作品，它是对 **“教育公平”** 和 **“因材施教”** 的科技响应。通过 Gemini 这种顶级多模态大模型，我们能让每一个普通工作坊的孩子都享受到“专属观察员”和“私人导师”级的服务，让流失的教育细节重新发光。
